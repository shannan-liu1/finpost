# 03 - End-to-end HTML runbook for RunPod

- **Status:** Ready
- **Ready for agent:** yes
- **Depends on:** 02-notebook-edits (the runbook references the canary cell and per-arm cleanup cells by name)

## Goal

A single standalone HTML file the user can open locally in a browser and read side-by-side with the JupyterLab notebook on RunPod. End-to-end: from picking a chip on the RunPod console through downloading eval results and stopping the pod. "Assume grandma" tone: every command explained, expected output shown, every failure mode flagged inline.

## Scope

**In scope:**
- Markdown source at `docs/runbooks/runpod-end-to-end.md` — authoritative version.
- Standalone HTML at `docs/runbooks/runpod-end-to-end.html` — **hand-written**, NOT generated by pandoc. Local Windows machine doesn't ship with pandoc, and adding a pandoc dependency to the project for a single doc is wrong proportionally. The HTML embeds a small `<style>` block in `<head>` (system-font stack, code blocks monospaced with subtle background, table borders, max-width body for readability). Same content as the markdown but rendered manually so the output is self-contained and fully under our control. Both files committed.
- Content covers the full lifecycle below.
- Consolidates relevant pieces from `docs/runbooks/runpod-bootstrap.md` and `docs/runbooks/phase1-training-runbook.md` so the user reads ONE document — the existing runbooks stay as topical references for other workflows but this document is self-contained for this run.

**Out of scope:**
- Multi-chip coverage. RTX 6000 / A40 only (48 GB VRAM). A "if you can't get 48 GB" sidebar links forward to the PRD's notes section.
- Pod recovery from a crashed mid-run. Out of scope — the canary + interleaved cleanup is the primary defense; recovery is a separate workstream.
- Linux primer. The runbook assumes `bash` exists and commands can be pasted — but explains what each one does.

## Sections

1. **What you'll do and what it costs.** One paragraph summary: "you'll spin up a 48 GB GPU pod on RunPod, run a 50-step canary (~1 minute, ~$0.01), then a 2000-step three-arm SFT run (~1 hour, ~$0.50), then eval (~30 min, ~$0.25), then download results and shut down. Total: ~$0.75–$1.10."
2. **Before you click anything on RunPod.** What to have ready: wandb token (optional; we use offline mode by default), HF account (only if you want to push checkpoints; optional), a local terminal for SCP/download.
3. **Pod creation, step by step.** RunPod console: pick template (PyTorch 2.4+ on CUDA 12.x), pick chip (RTX 6000 preferred; A40 fine; how to confirm), set Container Disk ≥ 40 GB, set Volume Disk if applicable, click Deploy. Screenshots-equivalent description (we can't embed real screenshots without setting them up, but text descriptions of the buttons).
4. **Connecting to the pod.** Prefer JupyterLab over web terminal (per `feedback_runpod_jupyterlab.md`). How to open it from the RunPod console.
5. **Terminal pre-flight.** `File > New > Terminal` in JupyterLab. The full ordered command list with expected output for each, and "if you see X, do Y" failure recipes. Same content as the notebook's pre-flight cell, in narrative form here.
6. **Notebook execution, cell by cell.** For each cell: what it does (1 sentence), what success looks like (the print or output to expect), what to do if it fails. Explicitly call out the canary cell as the gate that determines whether the run is safe.
7. **Mid-flow monitoring.** What to check at each `df -h` / `nvidia-smi` cell, what numbers are healthy, when to stop.
8. **After eval.** Tarball, download via JupyterLab file browser. Pod stop (billing pauses, volume persists) vs terminate (frees everything; only do this after files are downloaded). Recording final cost from `cost_summary.json`.
9. **Troubleshooting appendix.** Bulleted index of the failure modes covered inline above, with the section they live in. Lets the user jump to the right place when something goes wrong.

## Acceptance criteria

1. `docs/runbooks/runpod-end-to-end.md` exists and is at least 200 lines.
2. `docs/runbooks/runpod-end-to-end.html` exists and is non-empty.
3. Opening the HTML in a browser shows formatted content (headings render; code blocks are monospaced).
4. The HTML is self-contained (no external CSS/JS references that would fail to load offline). `grep -E 'href="https?://|src="https?://' docs/runbooks/runpod-end-to-end.html | wc -l` returns 0 (no external links in href/src). Hand-written with inline `<style>` block ensures this.
5. All 9 section headings above are present in the markdown.
6. The markdown contains the exact commands referenced in the notebook pre-flight (consistency check — if the notebook says `df -h /workspace`, the runbook says it too).
7. The markdown references `sft_phase1_runpod_ablation_2000.ipynb` (not the deleted `_3000` notebook).

Phase 1 Direct Preference Optimization Study

Runbook and implementation map for adding Direct Preference Optimization after the Qwen 0.5B supervised fine-tuning ablation. This is the offline preference track: one fixed pair dataset, one policy update, one honest comparison against SFT-best.

Last updated: 2026-05-18. Source workstream: .scratch/phase1-dpo-comparison/PRD.md. Operator notebook: notebooks/dpo_phase1_runpod.ipynb.

Decision Summary

GPU answer A single 48 GB GPU is the right target for the first full DPO run. A40 is okay. RTX A6000 is okay. RTX 6000 Ada is okay. Be careful with the name "RTX 6000": the older Quadro RTX 6000 is a 24 GB card and is not the default target for full fine-tuning.
  • Start DPO only after the SFT ablation chooses an SFT-best checkpoint.
  • Use held-out training prompts for preference pairs. Never use test prompts.
  • Policy model and reference model both start from the same SFT checkpoint.
  • Reference model is frozen; only the policy model is updated.
  • Evaluate Base, SFT-best, and SFT+DPO on the same exact-answer harness.
  • If DPO is not measurably better than SFT, say that and move on.

What DPO Is Testing

Direct Preference Optimization trains from triples: prompt, chosen response, and rejected response. The objective widens the margin between the policy model's preference for the chosen response and the rejected response, measured relative to a frozen reference model. 

loss = -log_sigmoid(
    beta * (
        (logp_policy(chosen | prompt) - logp_ref(chosen | prompt))
      - (logp_policy(rejected | prompt) - logp_ref(rejected | prompt))
    )
)

For this repo, the preference label comes from programmatic math grading: a correct final answer beats an incorrect final answer for the same prompt. Prompts where every sampled completion is correct or every sampled completion is incorrect produce no pair and are recorded as quality signals in the manifest.

Study Sequence

  1. Finish the RunPod SFT ablation and pick SFT-best from results/evals/ablation_a40_run_1/accuracy_summary.csv.
  2. Build a held-out training prompt set from GSM8K and MATH. This prompt set must be disjoint from the eval prompts.
  3. Generate K=8 completions per prompt from the SFT-best checkpoint at temperature 0.8.
  4. Grade every completion with the exact-answer verifier path used by the eval harness.
  5. Build pairs.jsonl by pairing correct completions against incorrect completions for the same prompt.
  6. Run the DPO loss parity test against a reference calculation before launching any trainer.
  7. Run TinyGPT DPO as the infrastructure canary.
  8. Run a 20-50 step Qwen DPO canary on the 48 GB GPU. Stop if loss is non-finite, pair batches do not fit, or throughput is unreasonable.
  9. Run the full DPO baseline from SFT-best.
  10. Convert the DPO checkpoint to Hugging Face format and evaluate Base, SFT-best, and SFT+DPO on the same held-out GSM8K and MATH prompts.

GPU VRAM Use for full Qwen 0.5B DPO? Notes
A40 48 GB Yes Best default on RunPod if price is reasonable. Ampere bf16 path is suitable.
RTX A6000 48 GB Yes Comparable memory class to A40; active workstation cooling.
RTX 6000 Ada 48 GB Yes Good choice if available. Newer Ada card, also 48 GB ECC.
Quadro RTX 6000 24 GB No for default full fine-tune Use only for tiny canaries, LoRA fallback, or reduced batch/sequence tests.
A100 / H100 40-80+ GB Yes Faster, but not required for Qwen 0.5B. Use only if price/time tradeoff wins.

The reason 48 GB is the target is not raw parameter size. Qwen 0.5B itself is small. DPO adds cost because each batch carries chosen and rejected responses and needs policy and reference log-probabilities. A conservative full fine-tune budget is:

Component Approximate memory Why it matters
Policy weights, bf16 ~1 GB Trainable model initialized from SFT-best.
Reference weights, bf16 ~1 GB Frozen model for reference log-probabilities.
Gradients ~1 GB Policy model only.
AdamW state ~4 GB Two fp32 moment buffers for policy parameters.
Activations and pair batches ~6-18 GB Depends on pair micro-batch, sequence length, and forward strategy.
Runtime overhead and fragmentation Several GB CUDA allocator, tokenizer/model buffers, temporary logits.
Initial DPO hyperparameter stance Start with max_seq_len=1024, bf16, pair micro-batch 2-4, gradient accumulation to recover effective batch size, beta=0.1, and learning rate in the 1e-6 to 5e-6 range. Increase pair batch only after the canary shows memory headroom and stable loss.

Stop / Go Gates

Gate Go condition Stop condition
Pair yield Enough mixed prompt groups to produce a useful pair set. Most prompts are all-correct or all-incorrect; adjust prompt set or K.
Loss parity Local DPO loss matches reference within 1e-5. Any disagreement before training. Fix math first.
Qwen canary 20-50 steps, finite loss, checkpoint written, no OOM. Non-finite loss, OOM, reference gradients, or missing checkpoint.
Full DPO eval SFT+DPO beats or ties SFT within clear confidence intervals. DPO hurts MATH materially or only improves parse rate without accuracy.

Throughput Optimizations

The current operator path now includes the low-risk throughput changes: larger Hugging Face generation batches with CUDA OOM halving, source-specific generation token caps, live notebook subprocess output, tokenized DPO pair caching, pinned-memory DataLoader transfer, and fused chosen/rejected model forwards. The heavier ideas below remain optional larger-sweep work.

1. vLLM for completion sampling (5-10× faster pair generation)

Where it helps. The pair-generation step inside scripts/build_dpo_pairs.py calls model.generate() from Hugging Face Transformers for 2000 prompts × 8 samples = 16,000 generations. This is the longest single step in the pipeline.

What vLLM is. An open-source inference engine from UC Berkeley's Sky Computing Lab that replaces transformers.generate() with a runtime designed around how autoregressive sampling actually behaves on a GPU.

Why it is faster. Three independent mechanisms stack:

  • PagedAttention. The KV cache (the running record of attention keys and values across already-generated tokens) is the memory bottleneck during sampling. Hugging Face allocates one contiguous slab per sequence sized for the worst case. vLLM borrows the operating system's virtual-memory idea: it splits the KV cache into fixed-size pages and allocates pages on demand. This frees memory for larger batches and removes the worst-case padding tax.
  • Continuous batching. Hugging Face runs a batch to completion; when one sequence finishes early (hits EOS), that GPU slot sits idle until the longest sequence in the batch finishes. vLLM swaps a new sequence into the freed slot immediately, so GPU utilization stays near the ceiling rather than draining as the batch unwinds.
  • Prefix sharing. For our K=8 case the same prompt generates 8 samples. vLLM caches the prompt's KV state once and forks at the sampling step. Hugging Face re-encodes the prompt 8 times.

Estimated speedup. Pair generation drops from roughly 10 minutes to 1-2 minutes at this scale.

Integration sketch. Replace the sample_completions function in scripts/build_dpo_pairs.py with a vLLM LLM(model=...) instance and call llm.generate once per prompt batch with SamplingParams(n=8, temperature=0.8, top_p=0.95, max_tokens=768). The grading path, JSONL writing, and pair construction stay identical.

Supply chain note vLLM itself was not compromised in the March 2026 PyPI supply chain incidents that affected litellm and telnyx. It is actively maintained by a healthy contributor base out of UC Berkeley. Pin an exact version (vllm==X.Y.Z) rather than a range, and verify the installed version against the official changelog before trusting it in a pipeline.

2. KV cache reuse via num_return_sequences (1.5-2× faster pair generation)

A no-new-dependency intermediate option if vLLM is too heavy. Today the flattened items list at the top of sample_completions repeats each prompt K=8 times in the batch, so the model re-encodes the prompt eight times: 

items = [(example, sample_idx)
         for example in examples
         for sample_idx in range(samples_per_prompt)]

Hugging Face's generate() accepts a num_return_sequences=K argument that runs the prompt encode once and forks K parallel decoders against the same KV cache. The refactor is to drop the duplication, pass one prompt per batch slot, and add num_return_sequences=K with a smaller outer batch width to keep the K-fold inner expansion within memory.

Smaller gain than vLLM (1.5-2× rather than 5-10×) because we still pay Hugging Face's contiguous KV slab and lack continuous batching, but it has zero dependency risk and is roughly a thirty-line change.

3. Precompute reference log-probabilities (~2× faster DPO training)

The DPO loss needs four log-probabilities per pair: logp_policy(chosen), logp_policy(rejected), logp_ref(chosen), and logp_ref(rejected). The reference model never updates. So the two reference quantities are static functions of (checkpoint, pair) and can be computed once, written next to pairs.jsonl, and loaded as a tensor during training.

With reference logprobs precomputed: each training step becomes a single forward pass per pair through the policy model only. The reference model can be deleted from GPU memory entirely after the precompute pass, freeing roughly 1 GB and letting the pair micro-batch grow from 2 to 4 or 8. End-to-end DPO training time drops by roughly 40-50%.

Implementation lives in finpost.training.dpo_train: set data.precompute_reference_logps: true and data.reference_logps_cache_path in the DPO config. Trainer setup iterates the tokenized pair dataset once under torch.no_grad(), stores per-pair log-probs as a torch cache keyed to the pair file, tokenizer, reference checkpoint, dtype, and sequence length, then frees the reference model. The training loop reads precomputed reference values instead of running the reference forward.

4. Skip the base eval inside the DPO notebook

The DPO evaluation cell currently runs three checkpoints: 

--checkpoints \
  base=Qwen/Qwen2.5-0.5B \
  sft_best=<sft path> \
  dpo=<dpo path>

The base model's numbers are already in results/evals/ablation_a40_run_1/accuracy_summary.csv from the SFT ablation. Re-running it costs about 3-5 minutes per DPO run for no new information. Two paths:

  • Quick path. Drop base=Qwen/Qwen2.5-0.5B from the eval CLI invocation in the notebook. After the DPO eval finishes, append the base row from the SFT ablation CSV when reading results for the comparison table.
  • Clean path. Teach finpost.evals.eval_exact a --reuse-from flag that accepts a prior accuracy_summary.csv and skips any (checkpoint, source) pair already measured at the same n and seed. This is more invariant-safe but requires schema discipline in the eval CLI.

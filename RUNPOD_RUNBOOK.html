Step-by-step guide for taking a local checkpoint through a RunPod A40 GPU session: upload to Hugging Face, provision pod, set up the workspace, smoke-test, run the real workload.

Based on the actual session that produced qwen_combined_hf_step_1000 + canary SFT run. Commands are copy-paste ready.

Cost framing
An A40 on-demand pod bills ~$0.44/hr = ~$0.007/minute. Setup + smoke + eval = ~$0.10. A full Phase 1 SFT run is ~$1. Every minute of debugging is real money — favour scp over slow web uploads, and stop the pod the moment you're done.

1.
Upload the checkpoint to Hugging Face
💻 Laptop

If your checkpoint is a .zip, unzip first — hf upload on a folder uploads each file individually; uploading the zip leaves you with a useless archive on the Hub.

If you have a zip

cd $HOME\Documents\GitHub\finpost\results
Expand-Archive -Path .\qwen_combined_hf_step_1000.zip -DestinationPath .\
ls .\qwen_combined_hf_step_1000

The ls should show at minimum config.json, tokenizer files (e.g. tokenizer.json), and model.safetensors.

Log in with a write-scoped token

hf auth login --force
# Paste your Write token. Answer 'n' to git credential.
hf auth whoami    # confirms your username (this is the namespace for uploads)

Upload the folder

hf upload <your-username>/finpost-sft-step-1000 `
    "$HOME\Documents\GitHub\finpost\results\qwen_combined_hf_step_1000"
Why HF Hub and not direct scp?
scp also works (see Gotchas). Hub is preferred when: you want to share the checkpoint across machines, you might re-use it in a different pod later, or your laptop has weird egress restrictions to arbitrary pod IPs. For a one-shot eval, scp is faster.

Pod name
Name it for the workload (e.g. eval-run-1 or sft-canary) rather than something generic. RunPod's history view sorts by name, and "Pod #4 from May 12" tells you nothing in two weeks.

4.
Clone the repo and install
☁️ Pod
cd /workspace
git clone https://github.com/shannan-liu1/finpost.git
cd finpost
git log --oneline -1

python -m venv .venv
source .venv/bin/activate
pip install --upgrade pip
pip install -e ".[dev]"

python -c "import torch, finpost; \
    print('finpost', finpost.__version__); \
    print('torch', torch.__version__); \
    print('cuda', torch.cuda.is_available()); \
    print(torch.cuda.get_device_name(0))"

The Python one-liner should print:

finpost 0.0.1
torch 2.11.0+cu130
cuda True
NVIDIA A40
Why "cu130" on a CUDA 12.4 host?
PyTorch wheels bundle their own CUDA runtime. pip's resolver picks the newest compatible build; cu130 works fine on a 12.4 host because the bundled runtime takes precedence. The cuda True + device name confirm it's wired up.
Install time
Expect 3–5 minutes for the first pip install -e ".[dev]" — torch alone is ~2 GB. Subsequent sessions on the same pod (with the volume disk) are much faster.
5.
Verify the CLIs exist (--help)
☁️ Pod

Cheap sanity check that the package imported correctly and the entry points work:

python -m finpost.training.train --help
python -m finpost.evals.eval_exact --help

Both should print a usage block. If either errors, fix the install before continuing.

6.
Run the tests
☁️ Pod

Optional but reassuring — confirms the trainer and checkpoint code paths work on this hardware. Takes ~2 minutes.

python -m pytest tests/test_config.py tests/test_dataset.py tests/test_optim.py \
    tests/test_checkpoint.py tests/test_trainer.py -q
Known failure on CUDA
test_checkpoint.py::test_save_then_load_round_trips_all_state fails with "Boolean value of Tensor with more than one value is ambiguous". This is a test-code bug — it uses Python == on a CUDA RNG state tensor instead of torch.equal(...). The same test uses torch.equal for the regular RNG state two lines up. Not blocking: the eval CLI doesn't touch this code path. It only matters if you intend to resume a training run from a saved RNG state. Tracked for a future fix.

Expected result: 45 passed, 1 failed in ~2 minutes. As long as the only failure is the one above, proceed.

7.
Mini smoke test — tiny-gpt2 on GPU
☁️ Pod

Proves the eval pipeline runs end-to-end on this pod's CUDA before you spend on the real workload. ~30 s, ~$0.004.

python -m finpost.evals.eval_exact \
    --checkpoints tiny=sshleifer/tiny-gpt2 \
    --sources gsm8k \
    --n 4 --seed 42 \
    --out-dir results/evals/_pod_smoke \
    --batch-size-gsm8k 2 \
    --gpu-cost-per-hour 0.44 \
    --device cuda

Verify:

cat results/evals/_pod_smoke/cost_summary.json

Sanity-check the JSON:

FieldExpect
gpu_type"NVIDIA A40"
dtype"torch.bfloat16" (A40 is Ampere; bf16 selected automatically)
generated_tokens > 0True
generated_tokens_decoded > 0True
estimated_cost_usd≈ elapsed_sec / 3600 × 0.44
Stop here if smoke fails
The real workload uses the same code path. If smoke crashes or produces garbage, fix it before spending real GPU time on the full run.
8.
Run the full workload
☁️ Pod

Two patterns shown — pick whichever matches your goal.

Pattern A — Evaluate base vs SFT (~10 min, ~$0.07)

First, pull the checkpoint you uploaded to HF Hub in step 1:

hf auth login
# Paste your token (read scope is enough here)
hf download <your-username>/finpost-sft-step-1000 \
    --local-dir results/checkpoints/qwen_combined_hf_step_1000
ls results/checkpoints/qwen_combined_hf_step_1000/

That ls must show config.json + tokenizer files + model.safetensors. Then run the eval:

python -m finpost.evals.eval_exact \
    --checkpoints base=Qwen/Qwen2.5-0.5B combined=results/checkpoints/qwen_combined_hf_step_1000 \
    --sources gsm8k math \
    --n 500 --seed 42 \
    --out-dir results/evals/base_vs_sft_run_1 \
    --batch-size-gsm8k 16 --batch-size-math 8 \
    --gpu-cost-per-hour 0.44 \
    --device cuda

Inspect:

cat results/evals/base_vs_sft_run_1/accuracy_summary.csv
cat results/evals/base_vs_sft_run_1/cost_summary.json
ls results/evals/base_vs_sft_run_1/

Expect six files: accuracy_summary.{json,csv}, two details_*.csv per checkpoint per source, run_metadata.json, cost_summary.json.

Pattern B — Canary SFT (~3 min, ~$0.02)

A 60-step training canary on the real model. Proves the training path works on this hardware before committing to a multi-hour run.

export WANDB_MODE=offline
python -m finpost.training.train \
    --config experiments/colab_qwen_500m.yaml \
    --device cuda \
    --max-steps 60

Healthy signs at end of run:

  • Final train/loss noticeably below initial (e.g. 0.78 vs. ~8 at step 0)
  • train/grad_norm finite (no NaN/Inf)
  • train/tokens_per_sec in the thousands (~4k on A40 for Qwen 0.5B)
  • Checkpoint written to results/checkpoints/<run-name>/step-00000060/
Canary checkpoint is NOT HF format
The trainer writes model.safetensors + state.pt — no config.json or tokenizer. The eval CLI needs HF format. To evaluate this checkpoint you'd need a conversion step (separate from this runbook). For the headline base-vs-SFT comparison, use a checkpoint that's already HF-formatted (e.g. one you uploaded via pattern A).

Pattern C — Full SFT training (1–3 h, ~$0.50–1.50)

Same as pattern B without the --max-steps override. Use the YAML's full max_steps. Tail the run via wandb (offline) or watch stdout. After it completes:

# If you want to keep it, push the trained checkpoint to HF Hub
hf upload <your-username>/finpost-sft-full-v1 \
    results/checkpoints/<run-name>/step-<final>/
9.
Pull results back & stop the pod
💻 Laptop
🌐 Console

Pull eval results (small, < 1 MB)

On your laptop. Get <port> and <pod-ip> from RunPod → Connect → SSH over exposed TCP.

mkdir $HOME\Documents\GitHub\finpost\results\evals -Force
scp -r -P <port> -i $HOME\.ssh\id_ed25519 `
    root@<pod-ip>:/workspace/finpost/results/evals/base_vs_sft_run_1 `
    $HOME\Documents\GitHub\finpost\results\evals\

Push trained checkpoints to HF Hub (if any)

On the pod:

hf upload <your-username>/finpost-sft-pod-v1 \
    results/checkpoints/<run-name>/step-<final>/

Stop or terminate the pod

In RunPod console:

  • Stop Pod — billing stops, volume disk persists. Use if you'll be back within a day or two.
  • Terminate Pod — releases everything (volume too). Use when fully done with this checkpoint.
Do not leave the pod running
Idle pods bill the full GPU rate. The biggest waste in this whole workflow is a forgotten pod overnight ($10+) — set a phone reminder if needed.

Gotchas observed in practice

SymptomWhat it meansFix
huggingface-cli: error: invalid choice: 'upload' huggingface_hub older than 0.20 pip install -U huggingface_hub; the binary is now hf, not huggingface-cli
Warning: huggingface-cli is deprecated. Use hf instead. CLI was renamed in huggingface_hub 1.0+ Use hf <subcommand>, not huggingface-cli <subcommand>
transformers requires huggingface-hub<1.0, but you have 1.14.0 Dep conflict on your laptop only Ignore — for uploads you only need hf. If it bothers you later: pip install -U transformers
403 Forbidden: You don't have the rights to create a model under the namespace "..." Either the namespace isn't your account, or your token is read-only hf auth whoami to see your username. Regenerate token with Write scope. hf auth login --force
httpx.ConnectError: WinError 10054 on hf auth login TLS connection killed mid-handshake (proxy / antivirus / ISP) Retry once. If persistent, switch to scp path — see below
warmup_steps (50) must be < max_steps (20) CLI --max-steps override violates the YAML's warmup setting Set --max-steps above the YAML's warmup_steps, or temporarily sed the YAML's warmup down
bash: ython: command not found Typo or arrow-key history navigation munged the line Retype the command. Avoid arrow-up history in xterm.js web terminals; copy-paste fresh
Eval refuses your training checkpoint Trainer writes raw state.pt + model.safetensors; eval needs config.json + tokenizer too Use a checkpoint that was saved in HF format (or write a conversion step)
Cannot paste in web terminal xterm.js uses non-standard paste shortcuts Ctrl+Shift+V (Win/Linux), Cmd+V (Mac), or right-click. Better: switch to SSH for real copy-paste

The scp alternative (if HF Hub keeps failing)

With SSH set up between laptop and pod, you skip the Hub entirely:

# On your laptop — replace <port> and <pod-ip> with values from
# RunPod console → Connect → SSH over exposed TCP
scp -P <port> -i $HOME\.ssh\id_ed25519 `
    "$HOME\Documents\GitHub\finpost\results\qwen_combined_hf_step_1000.zip" `
    root@<pod-ip>:/workspace/finpost/results/checkpoints/

Then on the pod:

cd /workspace/finpost/results/checkpoints
unzip qwen_combined_hf_step_1000.zip
ls qwen_combined_hf_step_1000/   # must show config.json + model.safetensors
When to prefer scp vs HF Hub
  • scp: single-machine round-trip, < ~5 GB, you have SSH set up
  • HF Hub: you'll re-use the checkpoint across multiple pods/sessions, you want to share it, or your laptop egress can't reach arbitrary RunPod IPs
